## V1

**HIGH · correctness · go-client** — `client/worktree.go:513`

**ReapWorktrees classifies live sessions from another namespace/cluster as orphans and deletes their worktrees**

- **Evidence:** Liveness comes solely from `states, lerr := c.backend.List(ctx)` (worktree.go:513) — the current client's namespace on the current kubeconfig context. Every dir under the shared <stateDir>/worktrees root not in that one listing is reaped: WIP-commit then `git worktree remove` + `os.RemoveAll(dir)` (worktree.go:593-594) + `c.index.Delete(id)` (worktree.go:595). The index entry's persisted Namespace (internal/index/index.go:82, written at Create, client/client.go:529) is never consulted — reapRepoRoot (worktree.go:615) reads only WorktreePath/RepoRoot/WorktreeBranch. Neither the archived worktree design nor TODO.md mentions cross-namespace/cluster reaping.
- **Scenario:** State dir is global (~/.local/share/sandbox, index.go:44-50) and shared across clusters. User has a live session on omni-prod, then runs `sandbox worktree gc` while KUBECONFIG points at the kind dev cluster (or with `-n other-ns` — worktree gc uses the --namespace-scoped newClient(), internal/cli/worktree.go:52). The prod session's id is absent from List, so its worktree is WIP-committed and removed while the session is running: the os.RemoveAll fallback permanently destroys gitignored files (.env, local caches) that `git add -A` never captured, the live session's mutagen alpha vanishes mid-sync, and the index entry recording the worktree is deleted.
- **Fix:** Before reaping, load the index entry and skip (report "skipped") any dir whose entry.Namespace != c.backend.Namespace(); treat index-less dirs as skip rather than reap-by-git-discovery, or gate them behind an explicit --force. Longer-term, record cluster identity (API host) in the entry and match on it too.
- **Verdict:** **CONFIRMED — FIXED 2026-07-18 (ceb573d)**: ownership gate in ReapWorktrees — an index entry is required and must record the current namespace; foreign-namespace and index-less dirs are skipped (and reported); the git-discovery reap path was removed. Longer-term cluster-identity (API host) matching deferred as noted.

**HIGH · correctness · go-sync** — `internal/cli/sync_support.go:84`

**MF5 auto-heal auto-resumes mutagen safety-halted sessions, allowing a halted mass deletion to propagate to the other side**

- **Evidence:** sync_support.go:83-85: `if st == syncpkg.SyncStalled { p.maybeHeal(id) }` -> maybeHeal runs `syncManager().Reconcile(ctx, string(id))` (line 119) = ResumeAll+FlushAll (internal/sync/sync.go:369-374). classify() maps every status containing "halted" — including mutagen's halted-on-root-emptied/halted-on-root-deletion safety halts — to SyncStalled (internal/sync/status.go:288-291, pinned by status_test.go:38 `{"halted", `[{"status":"Halted on root emptied"}]`, SyncStalled}`). Nothing distinguishes a safety halt from a transport stall.
- **Scenario:** Mid-session, the pod-side workspace root is emptied or deleted (agent runs rm -rf on the workspace, or a resumed pod comes up with an empty/replaced volume). Mutagen's safety mechanism halts the two-way-safe project sync specifically so the mass delete does NOT reach the laptop; `mutagen sync resume` is mutagen's documented confirm-and-propagate action for that halt. Within one dashboard probe cycle (+30s debounce) the prober reads SyncStalled and fires Reconcile -> ResumeAll, silently confirming the deletion — the local project tree is wiped. This defeats the M27 data-preservation rationale documented at sync.go:281-289 and the wave-2 empty-alpha delete-storm guard, which only protects the Connect path.
- **Fix:** Separate safety halts from heal-eligible stalls: in classify() (or in maybeHeal) detect statuses containing "halted" (root emptied/deletion/type-change) and surface them like conflicts requiring explicit user action; restrict the automatic Reconcile to error/problem transport stalls that resume cannot turn into a destructive confirmation.
- **Verdict:** **CONFIRMED — FIXED 2026-07-18 (43a15e2)**: 'halted' statuses split into SyncSafetyHalted ('halted (needs review)'), excluded from the MF5 self-heal via healEligible(); only transport stalls (and V14's paused-while-running) auto-heal. Pinned (status classification + heal gate).

**renderCitations renders web-controlled citation titles without the H4 control-sequence sanitization every other transcript surface applies**

- **Evidence:** renderCitations builds each footnote line as `lines = append(lines, dim.Render(truncate(fmt.Sprintf("  %d. %s", n, src), width)))` where src is c.Title/c.URL taken verbatim from the event. truncate (model_render.go:706-717) uses ansi.Truncate, which preserves ESC sequences and does not strip \r/\n or C0 controls. - **Fix:** Run each citation's title (and url) through the existing sanitizeToolOutput/stripNonSGR helpers (or at minimum collapseSpaces to flatten \r\n and strip controls) before the truncate in renderCitations; alternatively sanitize once in the runner's mapCitations so all consumers get clean strings.
- **Verdict:** **CONFIRMED — FIXED 2026-07-18** (landed with the citations change): `sanitizeCitationField` (all-ANSI strip + whitespace collapse + C0 strip) applied to title and url before composing the footnote line; pinned by `TestCitationFieldsSanitized`.

**MED · race · go-cli** — `internal/cli/sync_support.go:199`

**Index store adapters do full-entry load-modify-save outside Save's lock, so a concurrent CLI invocation regresses LastEventSeq/Snapshot/LastActivity**

- **Evidence:** indexTitleStore.SaveTitle: `entry, err := idx.Load(string(id))` then `entry.RenamedTitle = title; _ = idx.Save(string(id), entry)` — the whole on-disk entry is re-read outside index.Save's per-path lock and written back in full. Same pattern in SaveAutoTitle (:302), SaveAgentSessionID (:221), SaveSnapshot (:351), SaveDriver (:405). index.Save's merge (internal/index/index.go:227) only defers ZERO-valued fields to disk, so stale non-zero fields (LastEventSeq, LastActivity, Snapshot pointer) from the outer Load win over newer on-disk values. index.go:19-25 documents the lock as closing exactly this lost-update window, but only for callers that pass partial entries — these adapters don't.
- **Scenario:** A dashboard is attached and streaming (SaveSnapshot advances LastEventSeq to 5000). In another terminal the user runs `sandbox rename <id> foo` (or a second dashboard via `sandbox attach` snapshots the same session): rename's SaveTitle loaded the entry earlier at seq=100, then Saves — merge sees next.LastEventSeq=100 (non-zero → wins) and writes it, regressing the resume cursor and Snapshot. The next dashboard launch seeds rows from the stale snapshot (wrong status/usage) and replays thousands of already-seen events — the exact symptom the snapshot machinery was built to eliminate (index.go:100-106).
- **Fix:** Make the adapters construct partial entries (e.g. Entry{SandboxSessionID: id, SandboxName: id, RenamedTitle: title}) and let Save's locked load-merge fill everything else — that is the calling convention Save's doc comment (index.go:166-176) describes. For SaveSnapshot, which must overwrite Snapshot/LastEventSeq, add an Index.Update(id, fn) that runs the read-modify-write inside the per-path lock.
- **Verdict:** **CONFIRMED — FIXED 2026-07-18 (43a15e2)**: SaveTitle/SaveAutoTitle/SaveDriver write partial entries (Save's locked merge fills the rest); SaveAgentSessionID/SaveSnapshot use the new locked Index.Update read-modify-write. Concurrent snapshot-vs-title regression pinned.

**MED · race · go-client** — `client/session.go:375`

**Session.Close racing an in-flight Connect resurrects a dead connection and orphans the background sync goroutine**

- **Evidence:** Connect publishes state in stages with no closed-flag or generation guard: setHandles (session.go:358), setRunner (session.go:375), then startBackgroundSync stores a fresh bgCancel/syncTask (session.go:594-597). closeHandles (session.go:717-739) closes forwards and nils runner/bgCancel/syncTask, but nothing stops a Connect that is already past line 358 from re-publishing after it. - **Scenario:** Caller runs Connect in a goroutine (as the dashboard connector does) and calls sess.Close() during teardown while Connect is between PortForward and return. This also fixes the sibling leak where two concurrent Connects overwrite s.handles without closing the loser's forwards.
- **Verdict:** **CONFIRMED — FIXED 2026-07-18 (ceb573d)**: generation counter bumped by closeHandles; Connect re-checks under mu at every publish (handles/runner/background sync) and aborts superseded, closing its own forwards; also fixes the two-concurrent-Connects forward leak. -race pinned.
