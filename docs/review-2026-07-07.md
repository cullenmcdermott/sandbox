## A. Security (two agents, independent — converged on A1–A3)

### A1 — HIGH — Agent child processes inherit `RUNNER_TOKEN`; a prompt-injected agent self-approves its own permissions
`runner/src/claude.ts:222-231` builds SDK options with `env: { ...process.env, … }`
(same in the summarizer `claude.ts:815`, and the opencode child at
`opencode.ts:305-308`). So the spawned `claude` binary — and every Bash tool it
runs — inherits the pod's full env: `RUNNER_TOKEN` (the bearer token protecting the
whole runner API), `OPENCODE_SERVER_PASSWORD`, and the provider key. The `/exec`
path deliberately strips exactly these via `sanitizedExecEnv` (`exec.ts:26-43`), so
the *softer* escape (the SDK Bash tool the agent uses constantly) is the default
one. `guards.ts` matches only the literal string `\bANTHROPIC_API_KEY\b` — false
comfort: `env`, `printenv`, `set`, `cat /proc/self/environ`, `node -e …` all sail
past it.

Exploit (sharper than exfil): in default/acceptEdits mode a prompt-injected agent
runs one allowed Bash call —
`wget --header="Authorization: Bearer $RUNNER_TOKEN" --post-data='{"allow":true}' http://127.0.0.1:8787/sessions/$SANDBOX_SESSION_ID/permissions/<id>`
— and self-approves the permission it was just asked about, defeating the entire
approval flow. It can also start/interrupt turns and read the event log. The
provider OAuth token is separately exfiltratable over the example egress policy's
open 443-to-any-public (A3). **Three of the eight agents flagged this
independently.**

Fix: build the SDK (`claude`) child env from an explicit allowlist (reuse a
positive-allowlist `sanitizedExecEnv`); never spread `RUNNER_TOKEN` into it — the
claude runtime doesn't need it. The provider key must stay (the binary
authenticates with it); contain its exfil at the network layer (A3). Stop relying
on the `guards.ts` regex for secret protection.
**Correction (verified 2026-07-07):** for the `opencode serve` child you may
strip `RUNNER_TOKEN` but **not** `OPENCODE_SERVER_PASSWORD` — `opencode.ts:279`
hard-refuses to start without it (it's serve's own client-auth credential).
Containment of opencode's *in-agent* Bash therefore belongs at opencode's
tool-spawn / guardrail-plugin layer, not the serve env. (Also: only `wget` is
installed, not `curl`; `wget` alone suffices for the exploit.)

### A2 — LOW-MED — Event log + SSE stream persist tool inputs/commands/prompts verbatim, defeating the audit-log redaction
`runner/src/audit.ts` carefully runs `redactSecrets` before writing `audit.jsonl`
(M13), but `events.ts:191-224` `appendEvent` writes `JSON.stringify(payload)` to
`events.db` with **no** redaction and immediately broadcasts the same raw payload
over SSE. Payloads include `turn.started {prompt}`, Bash command strings, and
`tool.*`/`permission.*` inputs. A secret echoed in a command
(`curl -H "Authorization: Bearer …"`) lands cleartext on the PVC and replays to
every future `attach` client via `after=<seq>`. Single-tenant, so the defect is the
*inconsistency*: the audit redaction implies scrubbing that the richer, host-synced
event stream doesn't do. Fix: factor `redactSecrets` into a shared module and run it
in `appendEvent` before persist+broadcast (at least for `turn.started`, `tool.*`,
`permission.*`).

### A3 — INFO — Runner binds `0.0.0.0`; example egress allows 443 to any public host
`server.ts:77` `server.listen(PORT)` passes no host → binds all interfaces (same for
sshd:22 and opencode `serve --hostname 0.0.0.0`:4096). Containment rests on the
bearer token + a default-deny NetworkPolicy that no-ops without an enforcing CNI.
Independently, `k8s/networkpolicy-egress-allow.yaml:48-59` carves out RFC1918/CGNAT
but still permits 443 to all public IPs — the exfil channel that turns A1/A2 into
remote exfiltration. Mostly accepted/documented; for OSS, state prominently in
SECURITY.md that the 443-to-any example is an exfil channel and point at FQDN-scoped
egress (Cilium `toFQDNs`) for the provider hosts actually needed.

Checked clean (recorded): constant-time bearer compare gating every non-`/healthz`
route incl. SSE; all SQLite queries parameterized; 1 MiB body cap; SSE client cap
16; `after` rejects NaN/negative; per-session Secret with 256-bit `crypto/rand`
tokens; ed25519 SSH keys `0600`, session-id validated `^[a-z0-9-]+$` before
ssh-config interpolation; pod `AutomountServiceAccountToken:false`, drop-ALL caps,
`allowPrivilegeEscalation:false`, RuntimeDefault seccomp; sshd pubkey-only.

## B. Runner reliability (TypeScript deep review)

### B1 — MED — Missing `'error'` listener on the spawned `opencode serve` child crashes the whole runner
`opencode.ts:304-318` registers only `proc.on('exit', …)`. A spawn failure (ENOENT
`opencode`, EAGAIN) emits `'error'` on the ChildProcess; with no listener Node
throws an uncaught exception → the control-plane process dies (no `/healthz`,
`/idle`, event log). `exec.ts:120` handles this correctly; the supervisor doesn't.
Fix: add `proc.on('error', …)` that logs and schedules the same backoff respawn.

### B2 — MED — Single-active-turn 409 gate ignores observer-synthetic busy → concurrent prompts into a busy opencode session
`server.ts:169` checks only `reg.activeTurns.size > 0`. An interactive attach-driven
opencode turn never registers a turn — it's only `status:'busy'` via the observer
(`opencode-observer.ts:186`). So `POST /turns` is accepted mid-interactive-turn and
`opencode-turn.ts` prompts the *same* session concurrently; once the headless turn
registers, the observer suppresses all events, freezing its open cycle's mapper.
Fix: also 409 when `cfg.backend === 'opencode-server' && reg.state.status ===
'busy'` (mirror the interrupt route at `server.ts:213`).

### B3 — MED — `/exec` hangs past its 30s timeout and leaks grandchildren when a command backgrounds a child
`exec.ts:115-129` resolves on `child.on('close')` (fires only when stdio pipes
close) and the timeout SIGKILLs only the direct `bash`. `!sh -c 'sleep 300 &'` →
bash exits instantly, `sleep` inherits the stdout pipe, `'close'` doesn't fire for
300s → the HTTP response (and the TUI's `!cmd`) hangs for minutes despite
`timedOut=true`; grandchildren survive the kill. Fix: `spawn(…, {detached:true})` +
`process.kill(-child.pid, 'SIGKILL')`, or resolve on `'exit'` with best-effort
captured output.

### B4 — LOW-MED — An event that fails to persist is also dropped from the live SSE stream (contradicts R11's intent)
`events.ts:200-224`: on SQLite write failure `appendEvent` continues with `seq = 0`
and broadcasts ("a missed log event beats a killed turn"), but the broadcast filter
`evt.seq <= client.afterSeq` (`events.ts:271`) drops seq-0 for every client
(`afterSeq >= 0` always). A transient disk-full during `turn.completed` → event lost
from log *and* stream → TUI shows the turn working forever until reattach. (The
event-model agent flagged the same seq-0 hole from the consumer side — see D-note.)
Fix: bypass the filter for `seq === 0`, or assign a synthetic in-memory seq.

### B5 — LOW — `after` greater than current max seq is accepted and silently swallows live events
`server.ts:127-130` rejects only NaN/negative; `events.ts:271` then skips every live
event with `seq <= afterSeq`. If `events.db` is recreated (PVC replaced) while the
CLI index still holds the old high seq, reconnect replays nothing and live events are
filtered until seq climbs past the stale value. Fix: clamp `afterSeq` to
`lastSeq(sid)` (or 400 on `after > lastSeq`).

### B6 — LOW — Synchronous git calls block the event loop up to ~9s per turn boundary
`claude.ts:857-895` `emitWorkspaceStatus` runs three `execFileSync` git commands (3s
timeout each) at init and after every completed turn, on the single-threaded runner.
On a huge/cold repo all HTTP routes, SSE writes, and heartbeats stall. Fix: async
`execFile`, fire-and-forget.

### B7 — LOW — Corrupt `session.json` permanently crash-loops the pod
`session.ts:135-137` `JSON.parse(raw)` throws straight out of `main()`. A
zero-length/truncated file (rename-without-fsync after power loss) → CrashLoopBackOff
until someone execs into the PVC. Fix: catch, move the bad file aside, seed
`emptyState` with a loud log.

### B8 — LOW — Permission resolve races the auto-deny deadline and lies `resolved:true`
`server.ts:233-251` looks up the pending entry *before* `await readBody`; if the
deadline timer fires during that await (`claude.ts:454-469`), `pending.resolve()`
no-ops on the `settled` guard yet the route still returns `{resolved:true}` — the
client believes its allow landed while the tool was denied. Fix: have the resolve
callback report whether it settled; respond 409/410 when it didn't.

### B9 — LOW — Oversized/malformed bodies surface as 500, not 413/400
`httputil.ts:14-26` rejects with plain Errors; `server.ts:69-75` maps any rejection
to 500. A >1 MiB prompt or malformed JSON returns `500 {"error":"request body too
large"}` — a permanent client error dressed as transient, so retry loops treat it as
retryable. Fix: typed errors mapped to 413/400.

## C. Client / k8s / sync reliability (Go core)

### C1 — HIGH — Background observer connects leak SPDY port-forwards; nothing can close them
`internal/k8s/portforward.go:97` roots each forward at `context.Background()` ("closed
by the handle, not the caller ctx"), but `dashboard.ConnectResult`
(`connector.go:52`) exposes no close seam and `internal/cli/dashboard_connector.go`
calls `sess.Close()` only on connect *failure*. So `evictObserver` (`warm.go:105`)
claims to tear down the forward but only cancels the SSE ctx (`model_sse.go:609`) —
the forward goroutine, its listener, and its infinite reconnect loop survive; every
background stream reconnect (`model_sse.go:545-580`) spawns a fresh Session+forward
while the old one leaks; and `EventsPassive` failure (`model_sse.go:494-498`) leaks
the just-established forwards. Worst case: a leaked forward for a later-suspended
session enters `runForward`'s retry loop (`portforward.go:154-231`) — a pods List +
Sandbox Get every ≤10s forever. Directly undercuts the observer-cap work that landed
2026-07-06 (953ef87). Fix: add `Close func()` to `ConnectResult`/`CreateResult`
(backed by `sess.Close`) and call it from `cancelLiveSSE`/eviction/reconnect-
supersede; or reuse one Session per id like `ConnectResult.Reconnect` does.

### C2 — MED — `models.normalize` makes every non-Claude model lookup impossible
`internal/models/models.go:295-306` unconditionally prefixes non-`claude-` ids with
`claude-` and swaps dots for dashes, so `big-pickle`/`gpt-5.2` →
`claude-big-pickle`/`claude-gpt-5-2`, which can never match the `opencode`/`openai`
provider tables `canonicalProviders` explicitly ranks (`:134`). Every non-Anthropic
model silently gets `staticFallback` (200k ctx, $0) → wrong ctx% and cost for all
opencode sessions. Fix: try the raw lowercase id first; apply the claude-alias
normalization only as a second lookup.

### C3 — MED — Idempotent re-create with a different account reconciles the Secret but not the Sandbox env
On Secret `AlreadyExists`, `syncSessionCredential` (`backend.go:391,444-488`) patches
the credential + account label, but the Sandbox create at `:416` also hits
`AlreadyExists` and the pod template — which `buildEnv` (`:1433-1461`) baked the env
var *name* (oauth vs api-key) and *source* (per-session vs shared Secret) into at
first create — is never updated. Re-create an oauth session with a console account →
API key injected as `CLAUDE_CODE_OAUTH_TOKEN` → auth breaks silently; `SessionsForAccount`
also then misreports. Fix: on AlreadyExists also patch the Sandbox pod-template env,
or reject a re-create whose auth/account shape differs.

### C4 — MED-LOW — Observer connect to an opencode session opens 3 forwards instead of 1
`client/session.go:286-296`: `case opencode:` is tested before `case !full:`, so an
Observer connect (documented as port-forward + health only) to an opencode session
forwards HTTP+SSH+opencode; the latter two are never used (guarded by `if full &&
opencode` at `:394`). Pure SPDY/listener waste per background stream, tripling what
C1 leaks for opencode. Fix: order the switch `case !full` first.

### C5 — LOW-MED — ssh config values written unquoted; a state dir with spaces breaks all sync
`internal/sync/ssh.go:123` (`IdentityFile %s`) and `:134` (`Include ` + path)
interpolate paths unquoted. The default root is space-free, but `WithStateDir`'s own
doc (`client/client.go:118-124`) steers SDK consumers to `<appdir>/remote-sessions`
— on macOS `~/Library/Application Support/…` — where the config is invalid and every
mutagen sync fails obscurely. Fix: quote (`IdentityFile "%s"`, `Include "%s"`) or
reject a spaced state dir.

### C6 — LOW-MED — Connect's backgrounded `CreateInputs`/reaper work is unbounded
`client/session.go:458` `bgCtx` has no deadline; the first flush is bounded to 12s
(`:481`) but `CreateInputs` (`:506`, 7 mutagen execs) and `ensureReaperWithRetry`
(`:517`) run unbounded. `stagedRunner.StartTurn` gates every turn on `AwaitSync`
(`:184-192`), so a hung mutagen daemon → "prompt submitted, nothing happens" with no
advisory. Fix: wrap the background phase in a generous overall timeout (~60s) so
`task.finish` always runs.

### C7 — LOW — Create-rollback guard keyed to the Secret only; a pre-existing PVC can be deleted as collateral
`backend.go:297-307`: the defer skips rollback only when `secretPreexisted`. If the
PVC returns `AlreadyExists` (`:398-403`, a prior session's workspace) while the
Secret was freshly created, a later transient Sandbox-create failure rolls back all
three, destroying the pre-existing PVC the guard's comment promises to protect. Fix:
track `pvcPreexisted` and skip rollback when either pre-existed.

### C8 — LOW — Data race on `Session.projectPath`
`client/session.go:244-248` writes `s.projectPath` in Connect without `s.mu`, while
`ProjectPath()` (`:157`) reads it lock-free — neighboring fields (`fresh`, `handles`,
`runner`) are all mutex-guarded; this one was missed. Trips `-race`. Fix: guard it —
and cover the unguarded reads at `:234` (fresh-path State synthesis) and `:246` too,
not just the write site.

### C9 — LOW — `suspend`'s active-turn probe is unbounded (~40s) while `destroy`'s got a 5s cap
`internal/cli/commands.go:97` dials with the raw command ctx (5s/port ready-wait +
30s HTTP client) vs destroy's explicit `context.WithTimeout(ctx, 5s)` (`:204-212`).
Suspending a session on a half-dead node stalls ~40s. Fix: apply the same 5s probe
timeout.

### C10 — LOW — First `models.Limit` can synchronously fetch models.dev (5s) inside the TUI reducer
`internal/models/models.go:186-227` holds `p.mu` across the network fetch on first
use (cold or >24h cache); it's called from reducer paths (`session.go:477,503`,
`readmodel.go:116`) → the UI freezes up to 5s on the first `session.started`/usage
event of the day when models.dev is slow/offline. Fix: return the static fallback
immediately, refresh the table async. (Cross-refs C2 — same package.)

### C11 — LOW — `ConnectOptions.IdleTimeout` silently no-ops when a reaper already runs
`internal/k8s/reaper.go:71-74` `EnsureReaper` returns early if a non-finished Job
exists, so a reconnect with a different `IdleTimeout`/`ReaperImage` (documented as an
override, `client/session.go:84-85`) leaves the old flags in force with no signal.
Fix: compare the existing Job's args (or a spec-hash label) and replace on mismatch,
or document first-writer-wins.

Checked clean (recorded): SSE watchdog/forwarder decoupling + `after=<seq>`
contiguity (client added before synchronous replay — no gap/dup window); `cred`
manifest atomicity + Add rollback + keychain stdin-only secret passing; `index.Save`
locked merge; port-forward first-ready race; CreateSession errgroup + happens-before;
gitignore→`--ignore=` single-argv injection-safe; reaper decide/TOCTOU logic.

## D. Event-model parity (schema → mapper → renderer)

### D1 — HIGH — Tool completion is FIFO-matched (ignoring `toolUseId`) and open cards are never closed at turn boundaries
`transcript_reduce.go:72-92` `finishToolCard` pops `pendingTools[0]`; the comment
claims completions carry no tool name, but the runner **does** emit `toolUseId`
(`mapping.ts:236-245`, `opencode-turn.ts:296-298`) and the id→index map exists
(`flatTools`) yet is used only for start-dedup. Compounding: `pendingTools` is
cleared only by `/clear`; the `turn.completed`/`interrupted`/`failed` handlers call
`finalizeStreaming()` (text only) and no card closure, and neither backend emits
anything terminal for in-flight tools on abort. Scenario A: esc mid-Bash in turn 1
leaves the Bash card queued; turn 2's Read `tool.completed` FIFO-pops the stale Bash
card (shows Read's output as Bash's), Read spins forever — off-by-one cascades for
the session's life. Scenario B: parallel tools in one message complete out of start
order → outputs on wrong cards. Fix: resolve via `flatTools[ToolUseID]` first, FIFO
fallback; drain `pendingTools` (mark interrupted/failed) in the three terminal
handlers.

### D2 — HIGH — Mid-turn pod crash replays as "working forever"
`session.ts:93-95,123` coerces persisted `busy`→`idle` on boot but never appends a
terminal event for the orphaned turn; `index.ts:100-104` boot emits only
`session.started` (and `setStatus` fires only on a change *after* init). SIGTERM is
best-effort (500 ms, `index.ts:47-70`); a hard kill (OOM) emits nothing. On replay
the Go side drives `turn.started`→`StatusBusy` (`readmodel.go:189-191`) with nothing
to flip it back and deliberately lets `turnActive` survive catch-up
(`transcript_reduce.go:169-177`) → permanent spinner + working indicator after a pod
reboot. Distinct from TODO §7c (that exonerated replay on a *live* pod). Fix: when
boot flips busy→idle, append `turn.interrupted` (reason "runner restart") +
`session.status_changed{idle}` before the boot `session.started`.

### D3 — MED — All four `turn.*` payloads are off-schema (invisible to the drift gate)
`schema/events.json:63-187` defines no Turn payloads, yet the wire carries
`turn.started {prompt}`, `turn.completed {result,stopReason,numTurns,durationMs}`
(claude) vs `{}` (opencode), `turn.failed {message,subtype,errors}` vs `{message}`,
`turn.interrupted {reason}`. The Go TUI decodes `turn.failed` via `ErrorPayload`
(`transcript_reduce.go:322`) — works only because both shapes share a `message` key.
`schema_test.go` can't see any of it → silent drift. Fix: add Turn payloads to the
schema + `just gen` (also unblocks D5).

### D4 — MED — Interrupt mid-thinking leaves the live "Thinking" tail up forever and leaks the buffer
`m.reasoning` is cleared only by `reasoning.completed` (`transcript_reduce.go:536`);
`finalizeStreaming` doesn't touch it and no backend emits `reasoning.completed` on
abort. esc during a long think → status flips to needs-input but the live reasoning
tail keeps rendering (`transcript_list.go:109,225`). Fix: reset
`m.reasoning`/`reasoningBuf` in `finalizeStreaming`. **Correction (verified):** the
original "stale `reasoningBuf` poisons the next turn's empty `reasoning.completed`"
rationale is wrong — every `reasoning.completed` is preceded by a
`reasoning.started` that resets the buffer (`:512`), so no poisoning path exists.
The persistent-tail bug is real; the fix is still correct.

### D5 — MED — OpenCode transcripts have no user prompts on attach/replay
The adapter/observer map only assistant-role parts (`opencode-turn.ts:253-255`;
observer emits `turn.started {}` with no prompt) and never emit a `role:"user"`
message. The prompt lives only in the off-schema `turn.started` the transcript
ignores. Cold attach with `after=0` → a wall of assistant replies with no
interleaved questions (claude has parity via the user-echo path
`mapping.ts:209-213`). Fix: adapter emits `message.started/completed role:user` for
the prompt, or render `TurnStartedPayload.prompt` once D3 lands.

### D6 — MED — `tool.delta` carries no `toolUseId`/`parentToolUseId` → live input preview can attach to the wrong card
`mapping.ts:295-297` emits only `{partialJson}`; `handleStreamEvent` has
`parentToolUseId` in scope (`:254`) and drops it. The TUI, lacking an id, targets the
newest pending flat card (`transcript_reduce.go:553-573`), so a subagent's streaming
tool input animates onto a main-thread card's arg. Distinct from TODO §2b gap 1
(text/thinking deltas). Fix: thread `toolUseId` (from the enclosing
`content_block_start`) + `parentToolUseId` into `tool.delta`; target by id.

### D7 — LOW — Hook-blocked tools produce two `tool.failed` events; the second corrupts the FIFO
`claude.ts:302-306` (PreToolUse block) appends `tool.failed {tool:'Bash'}` with no
toolUseId; the SDK then delivers the blocked `tool_result(is_error)` → a second
`tool.failed {toolUseId, tool:""}`. The TUI pops the FIFO twice → mis-closes the next
card or renders an orphan with an empty label. Fix: drop the hook's synthetic emit
(the SDK result already surfaces it with an id), or fix matching per D1.

### D8 — LOW — Runner emits payloads that violate the schema's own `required` fields
`ToolPayload.tool` is `optional:false` (`events.json:100`) but
`tool.completed`/`failed` from `handleUserMessage` (`mapping.ts:234-246`) and
`tool.delta` (`:296`) omit it; nothing validates *emissions* against the schema (only
types). `ToolPayload.exitCode` is schematized (`:104`) but never emitted (exit codes
go only to `audit.jsonl`). Fix: mark `tool` optional (matching reality) or populate
it; exitCode cross-refs the §2c exit-code plumbing note.

### D9 — LOW — `session.State.Status` dual vocabulary in the public SDK
`Client.SessionState` (`internal/runner/client.go:174-188`) decodes the runner's
`status:'idle'|'busy'|'error'` into `session.State.Status`, whose enum is
`"RUNNING"/"SUSPENDED"/…` (`types.go:156-163`). Current callers read only
`ActiveTurnID`, but `Session.SessionState` is public SDK surface — a consumer
comparing `st.Status == session.StatusRunning` is silently always-false. Fix: a
distinct runner-status type, or normalize on decode. (Cross-refs §8 de-Claude break.)

### D10 — LOW — `TurnInput` mirror drift
(a) `Advisor` (`types.go:214-219`) is absent from TS `TurnRequestBody`
(`types.ts:82-106`) — a Go-side no-op but the TS mirror doesn't record it. (b)
`Resume` is typed `session.TurnID` (`types.go:196`) but the runner treats it as a
Claude session UUID / opencode session id (`claude.ts:129-134`) — an SDK consumer
following the type passes the wrong identifier. Fix folds into §8's `AgentSessionID`
rename.

### D11 — LOW — OpenCode errors outside a turn cycle are invisible; title events suppressed during headless turns
The observer feeds the mapper only while a cycle is open (`opencode-observer.ts:210`;
cycles open on assistant `message.updated`), so a `session.error` before any
assistant message (provider auth failure) maps to nothing — dashboard stays idle.
Separately, `activeTurnsSize() > 0` short-circuits everything incl. the
`session.title` passthrough (`:142-146` before the title branch at `:153`), so a
retitle during a headless turn is lost permanently. Fix: map pre-cycle
`session.error` to `turn.failed`/`error`; move the title passthrough ahead of the
suppression guard.

### D12 — LOW — Usage/cost semantics rough edges
`emitUsage` stamps `totalCostUsd:0` on every assistant message and failed turns
(`mapping.ts:343,347-366`) — a failed turn's real cost is dropped and successful
turns emit `usage.updated` twice back-to-back (`:321-328`, redundant rows).
`readmodel.go:145-149` applies cacheRead/Write only when `InputTokens>0`, so a
provider billing everything as cache (plausible for opencode, `opencode-turn.ts:403-409`)
never moves ctx%. Fix: guard the double-emit; apply cache tokens regardless of input.

Replay/dedup checked sound: both reducers dedup on seq, seq-0 markers pass
deliberately, autopilot re-drive is `!dup`-gated, append-before-stream + synchronous
`replayTo` before `: replay-complete` is reliable. The one seq-0 hole is B4.
Generated files (`eventtypes.gen.go`, `events.gen.ts`) match `schema/events.json`
exactly — no drift.

## E. Performance (two agents, independent — converged on E1/E4/E5-runner)

### E1 — HIGH — `tool.delta` handler re-parses the whole accumulated input JSON per delta (O(n²), often ×2)
`transcript_reduce.go:547-573`: per delta, `c.rawJSON += p.PartialJSON` (full copy)
then `json.Unmarshal` of the *entire* accumulated buffer — which fails mid-stream, so
it parses again with `rawJSON + "\"}"`. Plus `syncItems()` (O(blocks) rebuild) per
delta instead of the O(1) `streamDelta` path. A 100 KB file write in ~1 KB fragments
≈ 100 deltas × ~50 KB × 2 parses ≈ 10 MB of scanning + 10 MB of copies + 100 list
rebuilds per tool call, on the Update goroutine, multiplied across every warm model
via `ingest`. Fix: accumulate in a `strings.Builder`; throttle preview extraction
(parse on the work tick / every N deltas); replace `syncItems()` with a version bump
(mirror `streamDelta`).

### E2 — HIGH (runner) — SSE replay materializes the entire event log synchronously, blocking the runner event loop
`events.ts:277-282` `replayTo` / `:227-249` `readEventsAfter` use `.all()` →
materializes all `after=<seq>` rows (deltas persisted, retention default-off), calls
`JSON.parse` per payload, then `sseFrame` re-`JSON.stringify`s each (a pure round-trip
— the column is already JSON text), writing synchronously with no drain. A long
session (10⁵+ rows) attached with `after=0` (fresh laptop, `sandbox trace`) blocks
the single-threaded runner for the read+parse — stalling live turns, `/healthz`,
interrupts — and RSS spikes by ~the transcript size (OOM risk in a limited pod), per
connect. **Both perf agents flagged this.** Fix: `stmt.iterate()`; build frames by
splicing the raw `payload` column (no parse/stringify); pause on `res.write()===false`
until `'drain'`.

### E3 — MED (runner) — Live SSE broadcast has no backpressure
`events.ts:284-287` `writeSse` ignores `res.write()`'s return and there's no
per-client buffered-bytes cap. A half-open passive stream (dead port-forward, no RST
— the R6 failure mode) accumulates every event indefinitely → runner RSS grows MBs/min
for the hours TCP keepalive takes → OOM-kill presenting as "session died mid-turn."
**Both perf agents flagged this.** Fix: after write returns false, cap
`res.writableLength` (~4-8 MB) and destroy the socket past it — safe because reconnect
replays from `after=seq`.

### E4 — MED (runner) — Delta events persisted forever; log growth unbounded by default
`mapping.ts:289-297` writes every stream chunk as a SQLite row and `:160-168`
re-stores the full text on completion. Deltas are only needed for live tail + short
reconnect gaps (the TUI reconstructs final state from completed events,
`transcript_stream.go:17-26`), yet dominate the log forever, scaling E2's cost and
PVC growth. M34's all-or-nothing retention was rightly rejected; **delta-only
compaction** doesn't truncate replay history. **Both perf agents flagged this.** Fix:
on `turn.completed`, delete that turn's `*.delta` rows older than the last N turns
(seq gaps are fine — the client tracks `lastSeq` monotonic; remaining events stay
contiguous).

### E5 — MED — Dashboard passive SSE streams deliver one Update+View per event (no batching)
`model_sse.go:630-638` `liveSSENextCmd` reads exactly one event per `RunnerEventMsg`,
vs the foreground `waitForEvent` (`transcript_stream.go:156-180`) batching up to 512.
The runner broadcasts all events incl. deltas, so 3-5 busy warm sessions × ~30
deltas/s ≈ 100-150 full render pipelines/s (each O(sessions) + unmarshal + warm-model
`ingest` hitting E1) while the user views a static dashboard — the multiplier that
makes the §4 per-frame costs user-visible. **Both perf agents flagged this.** Fix:
mirror `waitForEvent`'s non-blocking drain (batch `RunnerEventMsg`, one attention/anim
check per batch); or runner-side, skip delta events for `passive=1` streams.

### E6 — MED — Live reasoning tail re-wraps the entire accumulated thinking text per frame
`transcript_render.go:354-363` `renderLiveReasoning`: the assistant tail got the
incremental stable-prefix cache; reasoning didn't. Every delta re-runs
`lipgloss.…Width(w).Render(text)` over the whole buffer — a full ANSI word-wrap. A
40 KB think re-wraps 40 KB per frame at the batch rate. Fix: cache the stable-prefix
wrap (reasoning is plain text — wrap only after the last committed newline), or slice
to the last N lines before wrapping (the §2c design caps at ~6 lines anyway).

### E7 — MED-LOW — Streaming tail re-hashes (and copies) the entire buffer per delta
`transcript_list.go:240-254` `ensureStreamTail`→`fnvStr`: the change key is
`fnv(entire assistantBuf)` per delta, and the string→`[]byte` allocates a full copy.
O(k·L/2) per message; a 100 KB message ≈ 250 MB transient allocs + hashing over the
turn, for the foreground tail and every warm session. GC churn lands as frame jitter
during the busiest streaming. Fix: running FNV updated with only the delta bytes, or
key on `(mode, epoch, buf.Len())` (length is monotonic within a tail's life).

### E8 — LOW — Go SSE consumer copies every event line twice
`internal/runner/client.go:403-422`: `scanner.Text()` allocates the full line, then
`[]byte(data)` re-copies for `json.Unmarshal`. A 100k-event replay at ~300 B ≈ 60 MB
avoidable allocs in a tight loop. Fix: `scanner.Bytes()` + `bytes.TrimPrefix` +
unmarshal from the slice (safe — `json.Unmarshal` copies into `RawMessage`).

### E9 — LOW — `appendEvent` re-prepares the INSERT statement on every event
`events.ts:201-207` (also `readEventsAfter:229`, `lastSeq:254`): better-sqlite3
doesn't cache prepared statements; `db.prepare(…)` re-parses SQL on the per-delta hot
path. **Both perf agents flagged this.** Fix: prepare once (lazily, keyed to the open
`db`), invalidate on `closeEventLog`/reopen.

### E10 — LOW-MED — Host-side event cache reopens the file per cached event
`internal/cli/sync_support.go:252-258` + `internal/index/cache.go:31-52`: every cached
event pays `newIndex()` + `MkdirAll` + `OpenFile(O_APPEND)` + write + close (~5
syscalls), from the foreground transcript *and* every warm model via `ingest`.
Separately `events.ndjson` has no cap → a months-old session's multi-hundred-MB file
makes cold attach memory-bound (`LoadCachedEvents` slurps it whole). Fix: hold an open
append handle per warm session; cap the cache (the transcript needs only enough tail
to render instantly — `events.db` stays authoritative).

### H1 — MED-HIGH — The observer cap (953ef87) is a no-op for the exact fleet §1d targets
`readmodel.go:180` maps runner `idle`→`StatusNeedsInput` and `:208`
`turn.completed`→`StatusNeedsInput`; `attention.go:17` counts NeedsInput as
`needsAttention`; `warm.go:42-48` `observerProtected` protects all attention rows;
and snapshot hydration restores NeedsInput *before* the admission loop
(`model_reduce.go:339`→`:373`). So NeedsInput is the steady state of every running

you get the ~30 API-server forwards the cap exists to prevent (compounding C1's
leak). The unit tests pass only because `runningSession()` uses `StatusIdle`
(`reconcile_test.go:12`) and the cap test seeds snapshot-less states. Fix: protect
only Waiting/Failed (+attached), or gate NeedsInput protection on unseen output
(`seenSeq < lastSeq`).

### H2 — MED — Evicting the only evictable class (Busy-mid-quiet-tool) kills detached autopilot with a false toast
Corollary of H1: Busy isn't protected, and "active turns keep sessions warm" fails
for a long quiet tool (no SSE between `tool.started`/`completed` → `touchObserver`

**"⟳ loop ended — session suspended"** (`app.go:930-938`), which is false. Fix:
protect Busy and/or `tr.autopilot.active()`/`queuedPrompt != ""` retained models;
or tear down only the stream (not the retained model — `after=lastSeq` replay
refreshes on refocus) and fix the toast wording.

### H3 — MED — Evicted-while-Busy row freezes a Busy spinner indefinitely
After eviction no `turn.completed` arrives; the watch maintains only lifecycle
`Running`, so the row keeps the SSE-derived Busy glyph until the user focuses it.
953ef87's message says this tradeoff is "recorded in TODO" — it is **not**. Fix: on
eviction stamp the row to a watch-derived status (like `degradeUnreachable`'s
default) or mark it visually cold.

### H4 — MED — Expanded tool-card output renders unsanitized control sequences into the frame (new exposure from 9e1d03b)
`kit.RemapANSI` passes non-SGR CSI verbatim (probe-verified: it only rewrites SGR,

strips whitespace), so this is new. Fix: strip `\r` and non-SGR escapes in
`clampOutputLines`/`toolExpandBody` (`transcript_render.go:536-540`).

### H5 — MED-LOW — Tabs break the two-line card's "never overflows" budget

### H6 — LOW — 64KB tool-output cap covers only the claude path
