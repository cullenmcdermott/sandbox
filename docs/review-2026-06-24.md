## Session startup latency

_I traced the full cold-start path for `sandbox claude` creating a brand-new session: runStartSession -> newBackend (kubeconfig) -> provisionSession (getcwd, newSessionID, ensureSSHKey, CreateSession = 3 serial k8s creates Secret/PVC/Sandbox, index Save) -> backend.Start (waitForPodReady, 2s poll, gated on image pull) -> RunAttached -> attachMsg -> connectCmd -> sessionConnector.establish(full=true): Status Get, Resume-skip, PortForward (serial HTTP then SSH, 5s ready-wait each), RunnerToken Get, waitHealthy (1s poll/3s timeout/30s budget), StageSync = startMutagen (CreateAll = 8 serial mutagen sync-create subprocesses) + a 12s BLOCKING initial flush, ensureReaper (Job Get+Create) — only then is the connection returned, the transcript built, and the initialPrompt submitted as the first SDK turn. Concurrently the dashboard Init seeds the list and fans out a background observer connect per running session (capped at 4), contending with the foreground connect. Headline conclusions, ordered by likely wall-clock savings: (1) the runner image pull dominates and nothing warms/prefetches it, and the default image carries an opencode-only npm layer the claude path doesn't need; (2) the 12s blocking first-sync flush needlessly gates the visible prompt and should be backgrounded; (3) several serial steps (3 k8s creates, 8 sync creates, 2 port-forwards) are independent and parallelizable; (4) a redundant Status Get and redundant ensureSSHKey sit on the hot path right after create/start; (5) ensureReaper and the background observer burst run synchronously/contend with the foreground connect and should be deferred/deprioritized; (6) the first SDK turn pays a full cold start serialized after everything. I did not compile; all findings are from reading the cited files._

### 🟠 MED — Image pull on waitForPodReady is the dominant cold-start cost with no warm/prefetch
- **Where:** `internal/k8s/backend.go:516`, `internal/k8s/backend.go:517`, `internal/cli/claude_remote.go:166`, `internal/cli/dashboard_connector.go:131`
- **Why:** backend.Start -> waitForPodReady polls every 2s (wait.PollUntilContextCancel(ctx, 2*time.Second, ...)). For a brand-new session the pod must be scheduled and the runner image pulled before isPodReady is true. The runner image is node:24-slim plus a global `npm install -g opencode-ai` layer (runner/Dockerfile) — a large image. Nothing in the repo warms/prefetches it: there is no DaemonSet image-puller, no pre-pull Job, and no WarmPool (CLAUDE.md explicitly notes the agent-sandbox SDK's WarmPool model was rejected). On a cold node this single step can be tens of seconds and is 100% serial in front of every other step (port-forward/health/sync all wait on Start returning).
- **Fix:** Cut perceived and real cold-start: (1) Pre-pull the runner image cluster-side — a tiny DaemonSet that runs `image: <runnerImage>` with a sleep/true command keeps it warm in every node's cache so a new pod skips the pull. (2) Shrink the image: the `npm install -g opencode-ai@1.17.7` layer (runner/Dockerfile:59) is only needed for opencode-server sessions — split into a claude-only image (default) and an opencode image so the common `sandbox claude` path pulls a much smaller image. (3) Tighten the readiness poll from 2s to ~500ms-1s so the moment the pod is actually ready is detected promptly (the 2s interval can add up to ~2s of dead wait after the pod is ready).

### 🟠 MED — 12s blocking initial-sync flush sits on the first-prompt critical path for every new session
- **Where:** `internal/cli/connect.go:142`, `internal/cli/connect.go:147`, `internal/cli/connect.go:157`, `internal/cli/connect.go:158`, `internal/sync/sync.go:185`
- **Why:** On the new-session path establish(full=true) runs startMutagen -> CreateAll (created=true for first-ever sync), then blocks on flushWithProgress under a 12s timeout (context.WithTimeout(ctx, 12*time.Second)). connect() does NOT return — and therefore attachReadyMsg / the transcript / the initialPrompt submission do NOT fire — until that flush finishes or times out. The comment acknowledges a healthy-but-large first sync 'may time out, which just means still uploading in the background'. So a large repo makes the user wait up to the full 12s staring at 'Syncing files...' before they can even see the prompt, even though the agent does not need the files staged to accept the prompt.
- **Fix:** Decouple first-prompt readiness from the initial flush. Return the healthy connection (and let the transcript open + accept the prompt) as soon as the runner is healthy, and run the bounded first flush in a background goroutine that feeds the existing per-session sync indicator (the same pattern already used for the reconnect branch at connect.go:175-179). Keep the warning surfacing for a genuinely broken transport, but stop gating the visible prompt on it. Alternatively shorten the block to a fast transport-proof (e.g. 2-3s) and let staging continue in the background.
- **Verifier note:** Treat this as a tunable-tradeoff, not a clean decouple, because the pod workspace is genuinely empty until mutagen stages files (sync.go:116) and the agent reads from that workspace. Options, in order of safety: (1) Keep blocking the first flush but lower the staging gate from 12s to a smaller, configurable budget (and/or surface a "files still uploading" banner in the transcript) — the visible-prompt readiness can come up while a bounded staging settle still runs. (2) Decouple the transcript open from staging (open the transcript + show the prompt input as soon as the runner is healthy, using the same backgrounded-flush pattern at connect.go:175-179), BUT keep the first turn's *submission* gated on staging completion: queue the initialPrompt and only POST the turn once the first flush/staging signals complete (or a timeout elapses with a visible warning). That preserves RV20/RV21's transport-proof + workspace-settle guarantees while removing the "staring at Syncing files before you can even see the UI" cost. Do NOT blindly let the agent start a turn before the first sync stages, or codebase-referencing prompts will run against an empty workspace.

### 🟠 MED — CreateSession does three creates and CreateAll does eight mutagen sync-create subprocesses, all serial
- **Where:** `internal/k8s/backend.go:226`, `internal/k8s/backend.go:252`, `internal/k8s/backend.go:260`, `internal/sync/sync.go:70`, `internal/sync/sync.go:131`, `internal/sync/sync.go:150`
- **Why:** CreateSession creates the Secret, then the PVC, then the Sandbox as three sequential API round-trips (each a separate Create call at backend.go:226/252/260). The PVC-first ordering is commented as intentional ('so it exists when the pod starts'), and the Sandbox references the Secret, but the Secret and PVC creation have no ordering dependency on each other. Separately, sync.CreateAll runs 1 project + 4 config-input + 3 transcript = 8 serial `mutagen sync create` subprocess execs (sync.go:118, loop at 131, loop at 150) on the connect path; process-spawn + mutagen daemon round-trip per call.
- **Fix:** Parallelize the independent k8s creates: fire the Secret and PVC creates concurrently (errgroup), then the Sandbox once both succeed — saves one API RTT on every new session. For CreateAll, the 8 sync-create calls are independent and could be issued concurrently (bounded errgroup), or batched; only the project sync is load-bearing for the first prompt, so the 7 config/transcript syncs could even be created lazily after the connection is handed back.

### 🟠 MED — First turn pays full SDK cold start; no warmup of query() while pod/sync are coming up
- **Where:** `runner/src/index.ts:107`, `runner/src/claude.ts:173`, `internal/tui/dashboard/transcript.go:503`, `internal/tui/dashboard/transcript.go:501`
- **Why:** The runner's main() just starts the HTTP server (index.ts:107) and does no SDK warmup; the first query() runs only when the first /turns POST arrives. On the client side the transcript submits initialPrompt as its first turn only after Init opens the event stream (transcript.go:501-506), which only happens after the full connect (health + 12s sync block) completes. So the SDK's first-query cold cost (model resolution, MCP/hook init in claude.ts:173+) is incurred fully serially after everything else, and the prompt isn't even sent until sync settles.
- **Fix:** Two levers: (1) Send the initialPrompt as soon as the runner is healthy (see the sync-decoupling finding) so the SDK cold start overlaps with any remaining sync staging rather than following it. (2) Optionally warm the SDK in the runner at boot — kick a no-op/cheap query() or pre-resolve the model/registry right after startServer() so the first real turn skips one-time init. Lower confidence on (2)'s payoff since SDK internals aren't visible here, but (1) is a clear win.

### 🟢 LOW — Redundant Status Get and redundant ensureSSHKey on the connect path immediately after create/start
- **Where:** `internal/cli/connect.go:93`, `internal/cli/connect.go:144`, `internal/cli/claude_remote.go:104`, `internal/cli/dashboard_connector.go:131`, `internal/cli/dashboard_connector.go:142`
- **Why:** For the new-session path the Creator (dashboard_connector.go) and runStartSession both call backend.Start (which Gets the Sandbox+pod every 2s poll until ready) and then immediately call sc.connect -> establish, whose very first action is another backend.Status(ctx, sc.ref) Get (connect.go:93) purely to branch on Gone/Suspended — impossible states for a pod that just passed waitForPodReady. Both are extra round-trips/IO on the hot path, not just redundant work.
- **Fix:** On the freshly-created path, skip the connect-time Status branch (the pod is known-running) — e.g. pass a hint into establish to bypass the Gone/Suspended check, or have the Creator call a connect variant that assumes running. Memoize the per-session SSH key path in the sessionConnector (or pass it from provisionSession) so establish doesn't re-derive it. Each saves an API/IO round-trip; minor individually but free.
- **Verifier note:** On the freshly-created path, have the Creator pass a hint into establish that both (a) bypasses the connect-time Status Get + Gone/Suspended branch (the pod is known-running) AND (b) supplies the already-known backend name (dashboard_connector.go has backendName), since st.Backend currently drives the opencode-vs-claude port-forward branch (connect.go:108) and is returned in the connection (connect.go:190) — dropping the Status Get without supplying backend would break that discriminator. Separately, memoize/thread the per-session SSH key path from provisionSession into the sessionConnector so establish (connect.go:144) doesn't re-read it from disk. Both are free micro-savings; treat as low priority since the new-session latency is dominated by Sandbox create + waitForPodReady polling, not these round-trips.

### 🟢 LOW — Port-forwards established serially (HTTP then SSH), each with its own ready-wait
- **Where:** `internal/k8s/portforward.go:47`, `internal/k8s/portforward.go:61`, `internal/k8s/portforward.go:90`, `internal/k8s/portforward.go:101`, `internal/cli/connect.go:122`
- **Why:** PortForward loops over the PortSpecs and calls forwardPort sequentially (portforward.go:47-57). Each forwardPort spins up its own SPDY dialer goroutine and blocks waiting for that forward's `ready` channel (up to 5s) before the loop moves to the next port. The full new-session connect forwards both runner HTTP (8787) and SSH (22) via ForwardSpecs (connect.go:122), so the SSH forward's establishment is fully serialized behind the HTTP forward's. Establishing a client-go SPDY forward involves an API-server round-trip to set up the stream, so two are ~2x one.
- **Fix:** Establish the forwards concurrently: start all forwardPort goroutines first, then wait on all their ready channels (errgroup / WaitGroup), closing all on any failure. Since each already runs its own goroutine and ready channel, this is a small refactor of the loop in PortForward and removes one SPDY-setup RTT from the critical path. Bonus: the runner HTTP forward is the only one needed before health/first-prompt; SSH is only needed for mutagen — it could even be established lazily alongside the (now-backgrounded) sync.

### 🟢 LOW — ensureReaper does a synchronous Job Get+Create on the connect path before the connection is returned
- **Where:** `internal/cli/connect.go:184`, `internal/cli/reap.go:75`, `internal/k8s/reaper.go:64`, `internal/k8s/reaper.go:82`
- **Why:** establish(full=true) calls ensureReaperForSession (connect.go:184) inline after the sync block and before returning the connection. EnsureReaper does a Jobs().Get then (if absent) Jobs().Create (reaper.go:64,82) — two API round-trips in the cross-namespace reaper namespace. This is pure idle-reaping bookkeeping with no bearing on the user being able to type the first prompt, yet it runs synchronously in front of the connection hand-back.
- **Fix:** Move ensureReaperForSession off the critical path — fire it in a detached goroutine (it already only logs a warning on failure) so the connection returns without waiting on two reaper-namespace API calls. The reaper starting a few hundred ms later has no correctness impact (idle timeout is minutes).
- **Verifier note:** Move ensureReaperForSession off the synchronous connect path, but do NOT reuse the connect ctx in the detached goroutine: app.go:1006 defers cancel() the moment establish returns, so capturing that ctx would cancel the in-flight jobs.Get/jobs.Create and leave the reaper unstarted. Fire it in a goroutine with its own fresh context.Background() plus a bounded timeout (e.g. 30s), then proceed to build/return conn. Since EnsureReaper is idempotent and only logs a warning on failure, the reaper starting a few hundred ms later has no correctness impact (idle timeout defaults to 15m). Note the win is modest: the common case is a single Get, and this is already far cheaper than the 12s sync flush and 30s opencode-ready wait on the same path, which is consistent with the low severity.

### 🟢 LOW — Launch burst fans out a background observer connect per running session; the SSE replay can be heavy and the cap may be loose
- **Where:** `internal/tui/dashboard/model.go:1316`, `internal/tui/dashboard/model.go:1321`, `internal/tui/dashboard/model.go:967`, `internal/tui/dashboard/model.go:936`, `internal/tui/dashboard/model.go:952`
- **Why:** applySeed starts a startLiveSSECmd for every running session not already streaming (model.go:1316-1322). Each does an observer connect (port-forward + health) gated by connectSem (cap maxConcurrentBackgroundConnects = 4, model.go:967). On the `sandbox claude` path this background fan-out runs concurrently with the foreground autoAttach connect, contending for the same kube-apiserver/SPDY setup and the same connectSem slots — i.e. the new session's own connect competes with N-1 observers' port-forward setups. The observer connect is already trimmed (runner-only forward, no sync) but it still streams events after=lastSeq, and a cold session with no cached snapshot resumes from seq 0 (full replay).
- **Fix:** Prioritize the foreground session: when launched via RunAttached/autoAttach, defer the background observer fan-out until after the foreground connect+attach completes (or give the foreground connect a dedicated path that doesn't share connectSem). Consider also: only auto-stream sessions that actually have unread/attention (others can stream lazily on first view), and confirm observers always resume from a cached lastSeq rather than 0 to avoid a full replay burst at launch. The cap of 4 is reasonable but is about steady-state stream count, not launch contention with the foreground connect.
- **Verifier note:** Keep the core idea (prioritize the foreground session at launch) but drop the inaccurate framing. Note that the foreground connectCmd (app.go:988) does NOT share connectSem — the contention is only at the kube-apiserver/SPDY port-forward layer, so the fix is about ordering, not semaphore slots. Optionally defer the applySeed background fan-out (model.go:1316-1322) until after the foreground attach completes when launched via RunAttached/autoAttach, e.g. gate the per-session startLiveSSECmd on a flag the foreground attach clears. The seq-0 concern is mostly moot in production (SnapshotStore is always wired and resumes from cached LastEventSeq); it only matters for a never-before-snapshotted session, so it is not worth a dedicated fix. Also weigh this against the deliberate warm-hide design (every running session stays warm) and the existing FU2 cap-4 throttle, which already address the steady-state burst.

## Performance

_I audited the TUI render hot paths (internal/tui/dashboard transcript.go, transcript_list.go, model.go render path, statusline.go, zones.go, tui/list/list.go), the warm-hide background machinery (warm.go, model.go syncPoll/SSE handlers), internal/sync polling, and the runner (events.ts SSE/sqlite, claude.ts per-turn fetches). I ran the existing scroll benchmarks and wrote one throwaway micro-benchmark to get concrete numbers. Headline NEW findings: (1) the dashboard's right-pane preview for a SELECTED warm session re-runs a full layout+reconcile+bodyView of the entire retained transcript every frame (~40µs/47KB/frame at ~30fps, O(total blocks)) with no unchanged-guard — model.go:2253→transcript.go tailLines→layout→reconcileItems; (2) the sync poll forks a `mutagen sync list` subprocess per warm session every 4s for the dashboard's whole life regardless of focus (model.go:679→sync_support.go→status.go→mutagen.go exec); (3) visibleSessions() re-filters+re-sorts+copies the session list 4+ times per frame with no memoization (groups.go:145-146 calls it twice in one statement). I confirmed the documented bodyView ~285µs/frame caveat (TODO.md:52) is still open via BenchmarkBodyView (283µs/241KB/56 allocs) — fitModal does two ANSI lipgloss.Width scans per visible line per frame. Lower-severity: the runner broadcast() re-serializes each event once per SSE client (events.ts:200), and the streaming-markdown safe-boundary predicates rescan the full growing buffer per delta (O(N^2) CPU, unguarded by the renderer-bytes sub-quadratic test)._

### 🔴 HIGH — Mutagen `sync list` subprocess forked per warm session every 4s for the dashboard's life
- **Where:** `internal/tui/dashboard/model.go:679`, `internal/tui/dashboard/model.go:111`, `internal/cli/sync_support.go:118`, `internal/sync/status.go:42`, `internal/sync/mutagen.go:28`
- **Why:** syncPollTickMsg (model.go:679) loops over ALL m.retained (warm) sessions every syncPollInterval = 4*time.Second (model.go:72) and fires probeSyncCmd(id) (model.go:111) + probeIdleCmd(id). probeSyncCmd calls m.syncProber, wired in the CLI to dashboardSyncProber (sync_support.go:118) → syncManager().StatusSummary (status.go:42) → m.r.Output('sync','list',...) which is exec.CommandContext(ctx, r.bin, args...) (mutagen.go:28) — a real fork+exec of the mutagen binary. So with N warm sessions the dashboard forks N mutagen processes (plus N HTTP Idle calls) every 4 seconds, unconditionally and forever, whether or not the user is even on the dashboard or looking at any of those sessions. The warm set can reach warmSoftLimit=12 (warm.go:14, not enforced) so this is ~12 process spawns + 12 HTTP round-trips every 4s = ~3/sec sustained.
- **Fix:** Only probe sync/idle for the SELECTED session (the only one whose SyncStatus/idle hint is rendered in renderDetailLines), and/or back off the interval and skip probing while the dashboard screen isn't focused. A single batched `mutagen sync list` (all labels) once per tick, parsed per-session, would replace N subprocess spawns with one.

### 🟠 MED — Warm-session detail preview re-lays-out and reconciles the entire retained transcript every dashboard frame
- **Where:** `internal/tui/dashboard/model.go:2253`, `internal/tui/dashboard/transcript.go:2008`, `internal/tui/dashboard/transcript.go:2025`, `internal/tui/dashboard/transcript.go:852`, `internal/tui/dashboard/transcript_list.go:180`
- **Why:** renderDetailLines() (called every dashboard frame via zones.go:157) ends with `if tr,ok := m.retainedTranscript(s.ID()); ok { tail := tr.tailLines(5,width) }` (model.go:2253). tailLines (transcript.go:2008) calls seedSize(width, max(n+4,8)) → layout() (transcript.go:2025/852), and layout() unconditionally calls m.syncBody() → reconcileItems() (transcript_list.go:180), which walks ALL m.items (O(total blocks)) re-checking fingerprints and calls m.body.SetItems(...) (rebuilds the survivors map + re-pins) — then bodyView() runs fitModal over the tail. There is no 'unchanged' guard: it redoes the full layout+reconcile every frame even when neither the size nor the blocks changed. I added a micro-benchmark mirroring the call (m.tailLines(5,60) on a 300-turn / 600-block warm model) and measured ~40µs/op, 47.5KB, 21 allocs PER FRAME. render()→renderZoned is driven by the animTickMsg loop at ~30fps whenever any session is busy (model.go:631-644, app.go:791), so the selected warm session pays this ~40µs + 47KB allocation 30×/sec continuously while an agent works — and it scales O(total blocks) with conversation length.
- **Fix:** Cache the tail render in the warm TranscriptModel keyed by (width, lastSeq/blocks-len): only re-run seedSize/layout/bodyView when the size or the block count changed since the last tailLines call. Alternatively have renderDetailLines memoize the preview lines per (selectedID, lastSeq, width) so an idle-but-busy-elsewhere dashboard doesn't rebuild a non-changing preview each frame. reconcileItems should also early-return when len(items)==len(blocks) and no item is dirty/fresh/mutable.

### 🟠 MED — bodyView still ~283µs/frame: fitModal does two ANSI-aware lipgloss.Width scans per visible line every frame
- **Where:** `internal/tui/dashboard/transcript_list.go:302`, `internal/tui/dashboard/transcript.go:777`, `internal/tui/dashboard/model.go:2381`, `TODO.md:52`
- **Why:** bodyView (transcript_list.go:290-307) calls fitModal(out, m.width-1, h) every frame. fitModal (transcript.go:777) does strings.Split, then for each of h lines (≈40 in a full transcript modal) calls lipgloss.Width(l) (an ANSI escape scan) and padRight(l,w) which itself calls lipgloss.Width again (model.go:2381). That's ~2×40 = 80 ANSI scans + a Split/Join per frame, on already-cached, in-width content. I re-ran BenchmarkBodyView (scroll_bench_test.go): 283173 ns/op, 241177 B/op, 56 allocs/op on a 300-turn transcript — matching the TODO.md:52 caveat 'bodyView still ~285µs/frame'. This runs at the work-tick / render cadence while attached. TODO.md:52-55 explicitly lists this as still-open ('Further wins possible by having tui/list emit width-padded lines or caching a running prefix-sum').

- **Verifier note:** Two complementary changes are needed; the line-width memoization alone only recovers ~half the per-frame cost. (1) In tui/list renderItemEntry (list.go:250-268), pre-pad each cached line to l.width (or store a parallel []int of per-line display widths) at cache-build time so list.Render emits already-padded lines; then fitModal can skip lipgloss.Width + padRight for in-width lines — this removes ~134µs/frame. Replace it with a manual per-line append of the 1-col scrollbar glyph (the body lines are already exactly w columns after step 1, so no width computation or lipgloss alignment is needed) — that turns the join into a cheap strings.Builder loop. Doing only (1) leaves bodyView at ~155µs; both together bring it toward the few-µs target. ### 🟠 MED — Dashboard re-filters and re-sorts the session list 4+ times per frame (visibleSessions has no memoization)
- **Where:** `internal/tui/dashboard/model.go:1486`, `internal/tui/dashboard/groups.go:145`, `internal/tui/dashboard/zones.go:157`, `internal/tui/dashboard/zones.go:186`, `internal/tui/dashboard/attention.go:24`

### 🟢 LOW — SSE broadcast re-serializes (JSON.stringify) each event once per connected client instead of once
- **Where:** `runner/src/events.ts:200`, `runner/src/events.ts:196`
- **Why:** broadcast() (events.ts:200) loops over all clients and calls writeSse(client.res, sseFrame(evt)) inside the loop; sseFrame (events.ts:196) does `data: ${JSON.stringify(evt)}\n\n`. ### 🟢 LOW — Streaming markdown safe-boundary predicates rescan the whole growing buffer per delta (O(N^2) CPU over a turn)
- **Where:** `internal/tui/dashboard/chat/streaming_markdown.go:121`, `internal/tui/dashboard/chat/streaming_markdown.go:176`, `internal/tui/dashboard/chat/streaming_markdown.go:188`, `internal/tui/dashboard/chat/streaming_markdown.go:233`, `internal/tui/dashboard/chat/streaming_markdown.go:111`

## Agent UX parity (Claude vs OpenCode)

_I compared the Claude and OpenCode backends end-to-end across the runner (runner/src/{agent,claude,opencode}.ts, server.ts, index.ts), the CLI (internal/cli/{root,claude_remote,commands,connect,shell}.go), and the TUI (internal/tui/dashboard/{app,model,session,external_pane,sort,attention,backend_picker,statusline}.go). The architecture is deliberately split: Claude is a first-class "native" agent driven entirely through the runner's normalized turn/SSE/event pipeline, while OpenCode is a supervised external process — the runner only spawns `opencode serve` (opencode.ts) and the local `opencode attach` client talks to it directly over a port-forward, embedded in the TUI as a raw PTY pane (external_pane.go). selectAgent returns null for opencode-server (agent.ts:47-48) and POST /turns 409s (server.ts:138-144), so OpenCode bypasses the entire normalized model. The consequence is consistent and intentional at the seam, but it cascades into many concrete UX divergences: OpenCode sessions emit no turn/permission/usage/tool/title events, so in the dashboard they are permanently stuck at the cluster-derived StatusIdle baseline — never busy/waiting/needs-input, never surfaced by attention routing, never showing ctx%/cost/unread/recent-tools/model in the detail pane. The runner's permission flow, Bash blocklist guard, and append-only audit log are Claude-only, so OpenCode's in-agent tool use is unaudited and ungated by the runner. CLI flags also diverge (no --model and no initial-prompt for opencode; cancel/suspend's turn-aware logic is inert for opencode). Branding/glyphs (theme/brand.go, backend_picker.go) ARE at parity. Most gaps are inherent to embedding a foreign TUI and are hard to close without a native OpenCode transcript; a few (audit/guard coverage, --model, status-row model label, docs) are closable._
